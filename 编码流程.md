### 编码流程（技能）

使用 FranxAgent 工具将想法转化为可维护代码的可靠工作流。在动手编写代码之前，先用清晰的规划明确每一步：要做什么、用到哪些库、逻辑如何流转。

#### 1. 了解全局
- 先读取项目结构：`read("./project-root")`，了解文件布局、模块职责和模块间依赖。
- 对关键文件调用 `read` 获取 AST 骨架和带行号的完整内容，以结构导航、以行号定位。
- 如果不能简明地向自己（或同伴）解释问题和解决思路，说明还需要更深入地理解上下文。

#### 2. 阅读源代码
- 利用 AST 骨架追踪调用链：从入口函数开始，按函数名定位到其定义所在文件及行号。
- 先关注结构（类、函数签名、模块划分），再深入具体实现。
- 阅读依赖或库的源码时，优先定位主模块，再根据需要查看非公开函数。
- 留意类型签名、接口抽象和导入关系——它们比注释更可靠地揭示设计意图。

#### 3. 详尽规划（最关键步骤）
- 明确要修改的文件、类和函数，以及大致行号范围。
- **用文字描述重构或新增功能的详细执行计划**，至少包含：
    - **目标**：要解决什么问题，预期达到什么效果？
    - **涉及文件列表**：每个文件要做什么改动（新增/修改/删除）。
    - **将要调用的外部库**：名称、简要用途，以及选择该库的简要理由。
    - **核心逻辑流程**：按步骤写出关键环节（可配合伪代码或流程图），覆盖条件分支、循环和数据变换。
    - **数据输入/输出**：函数接收什么参数，返回什么，可能抛出什么异常。
    - **边界情况及错误处理**：空值、超时、权限不足等场景如何应对。
- 针对每个逻辑步骤，可以用“TODO”作为临时注释骨架，然后逐一实现。

#### 4. 精确编辑（外科手术式）
- 使用 `write` 的 `edit` 模式，配合 `read` 提供的行号，只替换需要更改的行。
- **行号铁律（只在 `edit` 模式下，绝对不可违反）**：
    - **禁止计算！** 你的 `start_line` 和 `end_line` 必须是最近一次 `read` 操作中显示的原始行号，原封不动地复制过来，不做任何加减运算。
    - **禁止预测偏移！** 不要管“如果我多加几行，后续行号会不会变”。那是系统内部处理的事，与你完全无关。
    - **死记这个例子**：`read` 显示你要替换的行是 54 到 57，那你填的就是 `54` 和 `57`。永远别填 `59` 或任何你自己推出来的数字。
    - **执行前复检**：调用前必须念出这句咒语：“我使用的行号是 [X, Y]，这是我上一次 `read` 操作中的原始行号。”
- 添加新函数或类时，用 `edit` 在相邻函数之间精确插入，而非附加到文件末尾。
- 每次仅做一次逻辑变更，之后立刻用 `read` 刷新行号，确保后续编辑的行号仍然准确。
- 全新的文件则用 `write` 的 `overwrite` 模式直接生成。

#### 5. 注释、文档与版权声明
- **所有注释、文档字符串以及 Doxygen 标记域（如 @brief, @param, @returns）都使用英文。**
- 类和公共函数必须使用 **Doxygen 格式** 的文档注释。示例：
    ```cpp
    /**
     * @brief Brief description of the function.
     * @param param_name Description of the parameter.
     * @returns Description of the return value.
     * @throws std::runtime_error When something goes wrong.
     */
    ```
- **版权声明（版权头）的正确处理方式**：
    *   先检查项目根目录下的许可证文件：`read("./COPYING")` 或 `read("./LICENSE")`。
    *   如果许可证文件中包含 **"How to Apply These Terms to Your New Programs"**（或类似指导性字眼），则按照其中的说明，在**每个源文件头部**添加它所要求的版权声明头。
    *   **同时，观察项目内其他既有文件是如何处理的。** 如果项目中历史最久、最核心的几个文件都没有版权头，很可能表示该项目不需要。
    *   如果许可证文件**没有** "How to Apply" 部分（这常见于 MIT、BSD 等宽松许可证），则**不需要**在源文件中强制添加冗长的版权头。只需在项目根目录保留完整的 `LICENSE` 或 `COPYING` 文件即可。此时，若文件开头有简单的 SPDX 标识符（如 `// SPDX-License-Identifier: MIT`）是很好的补充，但不是强制要求。

#### 6. 逐一验证
- 每次编辑后立即用 `read` 确认变更内容：行的变化、插入位置是否正确。
- 使用 `command` 运行相关测试。若无测试，则编写可复现变更行为的最小脚本并执行。
- 出现 bug 时，先重新 `read` 相关代码逻辑，再修改，拒绝凭记忆猜测。
- 修复根因，而不是局部工作区补丁。

#### 7. 清理与沉淀
- 删除所有调试打印、被注释的旧代码和硬编码的测试值。
- 最终通读每个改动文件的完整内容，检查命名一致性、错误处理和风格统一。
- 如果变更过程中产生了可复用的知识，调用 `add_skill` 保存为技能，供后续复用。

#### 核心原则
- **先理解，再规划，最后写代码** — 顺序不可颠倒
- **详尽的计划可以避免一半以上的返工**
- **按行编辑，而非按文件** — 精准手术胜过全量重写
- **一步一验** — 小步骤、可追溯、可回滚
- **结构优先，细节其次** — 用 AST 骨架导航，再深入具体实现
- **追踪调用链** — 通过类型和函数签名理解模块协作
- **治本不治标** — 修复根因而非症状
- **注释即文档** — 用 Doxygen 格式编写可自动生成的 API 资料
- **遵循项目的既定许可习惯** — 查看许可证文件和既有代码，决定是否需要版权头
